# NORKA R35 — 3D Scroll Story

A Vite + React + strict TypeScript automotive storytelling experience using the real uploaded R35 GLB. The WebGL canvas is fixed full-screen while HTML sections scroll above it. One GSAP master timeline controls camera position, target, FOV, and section copy, so upward scrolling evaluates the exact same path in reverse.

The project is configured for **Yarn Classic 1.22.22 only**. `yarn.lock` is the single dependency lockfile.

## Requirements

- Node.js 20.19+ or 22.12+
- Yarn 1.22.22
- A browser with WebGL and hardware acceleration
- Non-commercial use because the supplied model is licensed CC BY-NC-SA 4.0

## Setup

```bash
yarn install --frozen-lockfile
yarn dev
```

Open the local URL printed by Vite.

## Production

```bash
yarn typecheck
yarn build
yarn preview
```

The production output is `dist/`. `vercel.json` is included; on Vercel use the Vite preset, build command `yarn build`, and output directory `dist`.

## Runtime assets

- `public/models/norka-r35-original.glb` — untouched reference copy of the uploaded GLB.
- `public/models/norka-r35-desktop.glb` — default web model with the supplied blue `paint.png` embedded into `ext_body` and geometry compressed with Meshopt.
- `public/hdr/automotive-studio.hdr` — local procedural studio HDR generated by `scripts/generate-studio-hdr.py`.

The web GLB has embedded textures and buffers. No external texture paths are created. Run `yarn inspect:model` to inspect the runtime model or `yarn inspect:model:original` to inspect the untouched reference. The visible footer attribution is read from `asset.extras` at runtime, with a defensive fallback only when metadata is absent.

The default runtime variant is `desktop`. Set `VITE_MODEL_VARIANT=original` only for source comparison, or generate the optional `mobile` artifact before selecting it.

## Scroll and camera architecture

`src/hooks/useScrollStory.ts` creates a single master ScrollTrigger in normal-motion mode. Its scrubbed timeline is the only writer to the story rig. Camera position uses two non-overlapping legs per section to form the orbit/raise/descend/pull-back path. Target and FOV span the complete section transition. Reverse scroll therefore evaluates the same timeline backward, with no separate camera animation on `onLeaveBack`.

`src/three/CameraRig.tsx` copies the mutable rig into the Three.js camera in `useFrame`; React state is not updated on every scroll frame. The projection matrix is updated only when FOV changes. Section text uses opacity and transforms.

For `prefers-reduced-motion`, the camera switches between static desktop/mobile shots and copy uses short opacity changes.

## Refined camera compositions

The camera shots were visually re-tuned without modifying the model, normalization, model scale, CSS transforms, waypoints, timeline architecture, or scene layout. The exact review viewports were:

- Desktop: `1440 × 900`
- Mobile: `390 × 844`

Only the `desktopShots` and `mobileShots` values in `src/three/cameraShots.ts` were changed. The before/after coordinates, measured framing notes, and constraints check are recorded in [`CAMERA_TUNING_REPORT.md`](./CAMERA_TUNING_REPORT.md).

## Explore mode

The final CTA disables story ScrollTriggers without resetting their visual state, locks the page at its current scroll offset, eases to the final Explore composition, then enables OrbitControls. Pan is disabled, zoom is bounded, and polar limits prevent moving below the floor. Exit disables controls, restores the final story shot, restores the same scroll offset, and re-enables ScrollTriggers without returning to the top.

## Development camera HUD

In development, press **D** to toggle the HUD. It shows camera position, target, FOV, current section, and master progress. **Copy shot** copies a valid `cameraShots.ts` object entry. The module is behind `import.meta.env.DEV` and is excluded from production output.

To show ScrollTrigger markers during development:

```bash
VITE_SCROLL_MARKERS=true yarn dev
```

## Camera tuning workflow

1. Run `yarn dev` and wait for the real GLB to reach 100%.
2. Press **D** to show the camera HUD.
3. Scroll to a section and inspect the actual composition.
4. Adjust only the matching entry inside `desktopShots` or `mobileShots` in `src/three/cameraShots.ts`.
5. Use **Copy shot** to capture position, target, and FOV.
6. Test both down-scroll and up-scroll. Target must continue tweening for the full section transition; do not introduce independent camera tweens.
7. Test portrait framing separately. Mobile shots are authored independently, not obtained by multiplying desktop coordinates.

## Model color restoration and compression

The original GLB is never overwritten. In the supplied archive, the `ext_body` texture embedded in the GLB is nearly white while the intended blue paint is supplied separately as `textures/paint.png`. The optimizer embeds the checked-in copy at `scripts/assets/norka-r35-paint.png` into the existing `ext_body` texture slot before compression.

The pipeline does not resize, convert, or recompress any texture, and it does not simplify, weld, join, or remove model geometry. It applies medium-level Meshopt compression with conservative quantization settings while retaining node/material names, material slots, PNG alpha, texture resolution, and topology.

Optimizer dependencies are pinned in `devDependencies`:

```bash
yarn install --frozen-lockfile
yarn optimize:model
```

Default output:

- `public/models/norka-r35-desktop.glb`

Run `node scripts/optimize-model.mjs --variant mobile` only when a separately named mobile artifact is needed. The runtime `useGLTF(..., true, true)` path includes Drei's Meshopt decoder.

The checked-in HDR is ready to use. To regenerate it, install Python 3 with NumPy and OpenCV, then run:

```bash
python -m pip install numpy opencv-python-headless
python scripts/generate-studio-hdr.py
```

## Performance and resilience

DPR is capped near 1.5 desktop, 1.15 mobile, and 1 on low-end mobile. Shadows and antialiasing are reduced on low-end devices. Rendering pauses while the tab is hidden. The scene avoids per-frame allocations, uses no heavy postprocessing on mobile, and provides a visible WebGL fallback.
